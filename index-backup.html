Prompt Engineering

How to talk to AI so it actually does what you want

The Core Formula
every great prompt = [ROLE] + [CONTEXT] + [TASK] + [FORMAT] + [CONSTRAINTS]
Kid version: Tell the AI WHO it is, WHAT you have, WHAT you want, HOW you want it back, and what it must NOT do.
Prompt Anatomy
PartWhat it doesExample
ROLESets expertise level"You are a senior .NET architect"
CONTEXTBackground info"This is a FastAPI app migrating to .NET 8"
TASKWhat to do"Rewrite the UserService"
FORMATOutput shape"Return a C# class with XML docs"
CONSTRAINTSHard limits"Don't change the public interface"
The 5 Levels of Prompt Quality
  • 1Vague: "Fix the bug" — useless
  • 2Specific: "Fix the null reference in UserService.cs line 47"
  • 3Contextual: Add WHY it's a problem + what you've tried
  • 4Constrained: Add what must NOT change
  • 5Expert: All above + desired output format + success criteria
Techniques
CHAIN OF THOUGHT (CoT)

Add "Think step by step before answering". Forces AI to reason before concluding. Use for complex logic.

FEW-SHOT EXAMPLES

Show 2–3 input/output pairs before your real question. AI learns your pattern and matches it exactly.

NEGATIVE EXAMPLES

Tell it what BAD output looks like: "Do NOT use synchronous calls like this: [bad example]"

XML TAGS FOR STRUCTURE
<context>Your project info</context>
<task>What to do</task>
<format>Output shape</format>

Claude handles XML-structured prompts better than free-text for complex tasks.

PLAN BEFORE ACT

"Before writing any code, write a plan. I'll approve it, then you execute." — Prevents runaway complexity.

PERSONA ASSIGNMENT

"Act as a strict security auditor with zero tolerance for vulnerabilities" gets completely different output than "review this code".

Do vs Don't
Give the AI a goal to solve
Give step-by-step instructions
Specify the output format explicitly
Assume it knows how you want it
Include what must NOT change
Only say what it should do
Ask it to explain before acting
Let it jump straight to code
Prompt Templates (Copy-Paste)
# BUG FIX
Context: [describe codebase/file]
Bug: [describe symptom + error message]
Tried: [what you already attempted]
Constraint: [what must not change]
Task: Find root cause, explain it,
then fix it. Run tests after.
# CODE REVIEW
Review [file] as a senior engineer.
Rate each issue: High/Med/Low
Fix High issues automatically.
List Med/Low as recommendations only.
Never praise code just to be nice.
# MIGRATION
Read [python file] completely.
Before writing code: state your plan.
Migrate to .NET 8 Clean Architecture.
Constraints: [list them]
After: run dotnet test. Report result.
When Prompts Fail
AI GOING IN CIRCLES

"Stop. Your last 3 attempts failed. What is fundamentally wrong with your approach? Propose a completely different strategy."

OVERENGINEERING

"That's too complex. What is the simplest possible solution? Simplicity > cleverness here."

WRONG OUTPUT FORMAT

"Your output format was wrong. Here is EXACTLY what I want: [paste example]. Now try again."

Context Engineering

How to structure knowledge so AI stays coherent across sessions

The Core Problem
AI has no memory between sessions. Every conversation starts blank. Context engineering is the art of deciding WHAT to load, WHEN to load it, and HOW MUCH to load — so the AI always has exactly what it needs.
context engineering = RIGHT info + RIGHT time + RIGHT amount
The 3-Tier Memory Model
From the arxiv paper: researcher built 108k line C# app solo using this exact system.
T1
HOT MEMORY — Constitution
Always loaded. Every session. SMALL. Only rules Claude must ALWAYS know: naming conventions, what never to do, critical architecture decisions.
T2
WARM MEMORY — Specialist Agents
Domain expert agents with deep embedded knowledge. Only activated when their domain is needed. Each carries its own sub-context.
T3
COLD MEMORY — Knowledge Base
Detailed spec docs. NOT loaded upfront. Retrieved on-demand when the agent needs specific knowledge. Scales infinitely.
Context Loading Strategies
StrategyWhen to useHow
ALWAYS-ONCore rules, never changesCLAUDE.md / constitution
ON-DEMANDModule-specific docsClaude reads spec files
TRIGGEREDTask-type detectionSkills with descriptions
INJECTEDDynamic data (DB, API)MCP server retrieval
SCOPEDOne agent, one domainSubagent with its own context
What Goes in Each Tier
TIER 1 — CLAUDE.md (keep it slim!)
INCLUDE: Naming conventions, banned patterns,
  critical architecture choices, tech stack
EXCLUDE: Module-specific logic, detailed specs,
  anything that changes often
TIER 2 — Agent Definitions
python-analyzer.md  ← reads Python
dotnet-writer.md    ← writes C#
test-expert.md      ← writes tests
reviewer.md         ← security/perf
db-expert.md        ← schema/queries
TIER 3 — Knowledge Base Docs
/specs/
  01-analysis.md      ← full app map
  02-module-user.md   ← UserService spec
  03-module-order.md  ← OrderService spec
  04-db-schema.md     ← all tables
  05-api-contracts.md ← endpoints
Context Window Management
COMPACTION — when to use

Run /compact after every major milestone. Context window at 80%+? Compact immediately or accuracy drops.

SCOPING — give exact paths

Instead of "look at the codebase", say "read only /src/services/user.py and /src/models/user.py". Precision saves tokens and improves accuracy.

SUBAGENT ISOLATION

Each subagent has its own fresh context window. Big tasks = use subagents so the main conversation stays clean.

THE BREVITY BIAS PROBLEM

Research finding: if you keep optimizing prompts for brevity, they collapse into vague generics. Specialist agents NEED detailed embedded knowledge to work reliably. Don't over-compress.

Context Anti-Patterns
Load specs on-demand from files
Paste 2000 lines into chat
Slim CLAUDE.md (core rules only)
Everything in CLAUDE.md
Subagents for domain isolation
One agent for everything
Update specs after learning
Repeat corrections every session
The Codified Context Loop
  • 1Observe — AI makes a mistake or asks a question
  • 2Capture — Write it into the right tier (T1/T2/T3)
  • 3Verify — Next session, did AI use the knowledge?
  • 4Refine — Too vague? Make it more specific
  • 5Repeat — Mistake never happens again
After 283 sessions, the researcher had 26,000 lines of context infrastructure. The AI never repeated the same mistake twice. That's the goal.
For Your Migration
# Minimal T1 CLAUDE.md
Tech: Python → .NET 8 Clean Architecture
Rule: NEVER modify Python source (read only)
Rule: API contract must stay identical
Rule: All new code needs tests
Rule: Use Result<T> not exceptions
# That's it. Keep T1 tiny.

# T3 docs created by Claude
"Analyze /python-app/ and save detailed
 specs for each module to /specs/

# Agents read specs, not the whole app
"Read /specs/user-module.md and migrate it."

Agentic AI Development

Building and using multi-agent systems that actually work

What Makes Something "Agentic"
TypeWhat it doesExample
LLM APPAnswers prompts"Explain this code"
SINGLE AGENTHas goal + tools + loopClaude Code solo task
MULTI-AGENTCoordinated specialistsMigration pipeline
SWARMParallel autonomous workers6 modules at once
Agentic = the AI decides its own steps to reach your goal. You give the destination, it finds the route.
The Agent Loop (OODA)
every agent continuously does: Observe → Orient → Decide → Act → repeat 
Observe:  reads files, runs commands, gets output
Orient:   understands what it sees in context of goal
Decide:   picks next best action
Act:      writes file / runs command / calls API
Repeat:   until goal is complete or it asks you
4 Core Agent Patterns
1. ASSEMBLY LINE — sequential pipeline
Analyzer → Writer → Reviewer → Fixer → Tester
Each step reads prev step's output file.
Best for: tasks with clear dependencies.
2. COMMITTEE — parallel reviewers
Security Agent  ┐
Performance Agent├→ Aggregator → Report
Architecture Agent┘
Best for: quality gates, reviews, decisions.
3. PARALLEL SPRINT — independent workers
Agent A: UserService   ┐
Agent B: OrderService  ├→ all at once
Agent C: ProductService┘
Best for: independent tasks, max speed.
4. ORCHESTRATOR + WORKERS
You → Orchestrator → delegates → Workers
              ↑                         |
              └──── collects results ───┘
Best for: complex unknown tasks.
Orchestrator plans dynamically.
Coordination via Files (Key Insight)
Agents don't talk to each other directly. They communicate via files. Agent A writes /specs/user.md. Agent B reads /specs/user.md. This is robust, inspectable, and recoverable. 
# Prompt that orchestrates via files
"Step 1: Read /python-app/user.py
  Save analysis to /specs/user-spec.md

Step 2: When Step 1 is done, read
  /specs/user-spec.md and write .NET code
  Save to /dotnet-app/User/

Step 3: When Step 2 is done, read both
  original Python and new .NET code.
  Write tests. Run dotnet test."
Tool Types Every Agent Needs
ToolWhat it gives the agent
read_fileRead any file in the project
write_fileCreate or update files
run_commandExecute shell commands, run tests
search_codeFind patterns across codebase
web_searchLook up docs, APIs, solutions
MCP toolsGitHub, Slack, DB, custom APIs
Agent Failure Modes + Fixes
HALLUCINATION — invents APIs/functions

Fix: Add to T1 "Before using any API, verify it exists by reading the docs or the codebase." + give it search tools.

CONTEXT DRIFT — forgets earlier decisions

Fix: Write decisions to a file immediately. "Save this decision to /specs/decisions.md". Agent reads it next time.

SCOPE CREEP — changes things you didn't ask

Fix: Be explicit in constraints. "Only modify files in /src/services/. Touch nothing else."

REPEATED MISTAKES — same error new session

Fix: Codified Context loop — capture the mistake in T1 or T3 immediately. Never repeat the explanation.

The Mental Shift
Think: "What outcome do I need?"
Think: "What steps should it take?"
Delegate the problem
Delegate the solution
You are the architect/reviewer
You are the typist
The expert move: Spend 80% of your time on the PROMPT and 20% reviewing output. Not the other way around.

Part 2 — The .claude/ Architecture

Project-Level · User-Level · Core Components (Rules, Skills, Hooks, Agents)

Project-Level (your-project/)

Shared in repo — team-wide configuration

Overview
Project-level config lives inside your repo. It's committed to Git and shared with your entire team. This is where you define coding standards, agent behaviors, and project-specific rules that every contributor should follow.
Directory Structure
📁 your-project/
├── CLAUDE.md           ← root constitution
├── 📁 .claude/
│   ├── CLAUDE.md       ← additional rules
│   ├── settings.json    ← project settings
│   ├── settings.local.json ← gitignored!
│   ├── 📁 rules/       ← always loaded
│   │   ├── code-style.md
│   │   ├── testing.md
│   │   └── api-design.md
│   ├── 📁 skills/      ← on demand
│   │   └── 📁 <skill-name>/
│   │       ├── SKILL.md
│   │       ├── reference.md
│   │       ├── examples/
│   │       └── scripts/helper.sh
│   ├── 📁 agents/
│   │   └── <agent-name>.md
│   ├── 📁 hooks/
│   │   └── protect-files.sh
│   └── 📁 commands/     ← legacy
CLAUDE.md — The Constitution
ROOT CLAUDE.md (your-project/CLAUDE.md)

The main instructions file. Always loaded into context at the start of every session. Keep it slim — only core rules that apply to the entire project.

.claude/CLAUDE.md

Additional project-level instructions. Also always loaded. Use this for supplementary rules that would clutter the root file.

# Example CLAUDE.md
Tech Stack: Python 3.12 + FastAPI + PostgreSQL
Style: Black formatter, 88-char lines
Testing: pytest, >80% coverage required
Rule: NEVER commit secrets or API keys
Rule: All public functions need docstrings
Rule: Use Result types, not bare exceptions
Settings Files
settings.json — Shared Project Settings

Committed to repo. Controls project-wide behavior: allowed tools, model preferences, permissions, MCP server configs shared by the team.

settings.local.json — Personal Overrides

GITIGNORED Personal overrides that should NOT be shared. API keys, local paths, individual preferences.

{
  "model": "claude-opus-4-6",
  "permissions": {
    "allow": ["read_file", "write_file",
              "run_command"],
    "deny": ["web_search"]
  },
  "mcpServers": { ... }
}

User-Level (~/.claude/)

Personal setup — follows you across all projects

Overview
User-level config lives in your home directory. These are YOUR personal preferences — coding style, favorite patterns, common mistakes to avoid. They apply to every project you work on and are never shared with the team.
Directory Structure
📁 ~/.claude/
├── CLAUDE.md           ← personal rules
├── settings.json       ← global settings
├── 📁 rules/           ← personal rules
│   └── *.md
├── 📁 skills/
│   └── <skill-name>/SKILL.md
├── 📁 agents/
│   └── <agent-name>.md
└── 📁 projects/        ← auto-memory
    └── <project-hash>/
        └── memory/
            ├── MEMORY.md
            └── topic-files.md
Personal CLAUDE.md
~/.claude/CLAUDE.md

Your personal constitution. Applied to every project, every session. Keep preferences that follow you everywhere — like your preferred error handling style or documentation habits.

# ~/.claude/CLAUDE.md
Preferences:
- I prefer explicit error handling
- Always use type hints in Python
- I like concise commit messages
- Explain complex logic with comments

Common Mistakes I Make:
- I often forget to close DB connections
- Remind me to add input validation

Style:
- Prefer functional style over OOP when possible
- Use f-strings, not .format()
Auto-Memory System
Claude remembers across sessions. The projects/ directory stores per-project memory that Claude writes automatically. It captures decisions, patterns, and learnings so they persist between conversations.
MEMORY.md — Auto-generated

Claude writes notes here during sessions. Architecture decisions, discovered patterns, things that worked or failed. Loaded automatically next session.

Topic Files — Specialized Memory

Organized by topic (e.g., debugging.md, migrations.md). More granular than MEMORY.md. Claude creates these as knowledge accumulates.

Config Precedence
loading order (highest priority last): User → Project → .claude/ → Local Overrides
  • 1~/.claude/CLAUDE.md — personal defaults (lowest priority)
  • 2your-project/CLAUDE.md — project constitution
  • 3.claude/CLAUDE.md — additional project rules
  • 4.claude/rules/*.md — always-loaded rules
  • 5.claude/settings.local.json — personal overrides (highest)
Conflict resolution: Project-level settings override user-level. Local overrides override everything. More specific always wins.

Core Components

Rules · Skills · Hooks · Agents — the building blocks

The Four Pillars
R
📋 RULES — Always Loaded
Markdown files in rules/. Loaded into every session automatically. Your permanent coding standards and conventions.
S
📦 SKILLS — On Demand
Domain-specific knowledge packages. Triggered by matching task descriptions. Contains SKILL.md, reference docs, examples, and scripts.
H
⚙️ HOOKS — Event-Driven
Scripts that run automatically on events (pre-commit, file-save, etc.). Configured via settings.json. Automate safety checks and workflows.
A
👤 AGENTS — Specialized Roles
Custom agent definitions with specific expertise, tool restrictions, and embedded domain knowledge. Invoked by name or auto-matched.
Skills — Deep Dive
SKILL STRUCTURE
📁 .claude/skills/<skill-name>/
├── SKILL.md       ← entry point + metadata
├── reference.md   ← detailed docs
├── examples/      ← code examples
└── scripts/       ← helper scripts
# SKILL.md — frontmatter
---
name: database-migration
description: |
  Use when creating or modifying database
  schemas, running migrations, or writing
  complex SQL queries.
applyTo: "**/*migration*,**/models/**"
---

You are a database migration expert.
Always use Alembic for Python projects.
Include rollback steps in every migration.
Test with both empty and populated DBs.
Skills vs Rules: Rules are always loaded (small, general). Skills are loaded on-demand (can be large, domain-specific). Use skills when the knowledge is only needed sometimes.
Hooks — Deep Dive
HOOK CONFIGURATION

Hooks are shell scripts triggered by events. Configure them in settings.json under the "hooks" key.

// .claude/settings.json
{
  "hooks": {
    "preCommit": [
      ".claude/hooks/protect-files.sh",
      ".claude/hooks/run-linter.sh"
    ],
    "postSave": [
      ".claude/hooks/auto-format.sh"
    ]
  }
}
# .claude/hooks/protect-files.sh
#!/bin/bash
# Prevent modifications to critical files
PROTECTED="package-lock.json .env.production"
for file in $PROTECTED; do
  if git diff --cached --name-only | \
     grep -q "$file"; then
    echo "❌ Cannot modify $file"
    exit 1
  fi
done
